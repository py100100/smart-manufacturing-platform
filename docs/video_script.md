# 项目讲解视频逐字稿

本文档为项目讲解视频录制逐字稿，可直接按顺序朗读。录制前请先运行：

```powershell
.\start-dev.bat
```

然后打开启动窗口中打印的 `Frontend` 地址。

## 1. 开场介绍

大家好，我现在讲解的是“基于 Agentic RAG 与多智能体协作的智能制造服务平台”。

这个项目面向真实工厂管理场景，目标不是做一个简单聊天机器人，而是做一个可以辅助生产、质量、设备、供应链和工艺团队协同工作的企业级智能运营后台。

在传统工厂里，生产排程、质量缺陷、设备故障、物料缺料和工艺参数优化通常分散在不同部门。问题出现时，需要人工跨部门沟通，信息链路长、响应慢，而且很多数据没有形成闭环。

所以本项目把工厂管理拆成 5 个独立智能体，每个智能体负责一个明确业务域，同时通过统一编排器完成自动路由和多智能体协同。每次智能体执行后，系统都会输出摘要、决策、证据、后续行动和节点反馈，方便前端展示和后续追溯。

## 2. 技术栈说明

本项目后端使用 FastAPI，负责提供智能体接口、编排接口和健康检查接口。

数据库层使用 MySQL，通过 SQLAlchemy 和 PyMySQL 连接。数据库模块只负责连接和初始化，不在接口层堆业务逻辑。

Agent 和 RAG 核心使用 LangChain 和 Chroma。知识检索层优先使用 LangChain 加 Chroma 做向量检索；如果依赖不可用，会自动回退到 TF-IDF 检索，保证演示环境稳定可运行。

大模型调用层预留了 DeepSeek API 接入，同时系统也支持本地规则引擎兜底。也就是说，即使没有配置外部 API Key，项目依然可以完成智能体流程演示。

为了避免智能体只返回“已接收请求”这类空洞话术，系统增加了 `BusinessAnswerService` 业务答案增强服务。它会根据问题所属业务域生成结构化专家答案；如果模型不可用，就使用本地专家模板兜底。编排器会对顶层 `summary` 做质量检查，必要时追加“答案综合”节点。

前端使用 Vue3、Vite 和 Element Plus，做成企业级运营后台，包括登录注册、运营总览、智能体工作台、生产调度看板、质量检测中心、设备监控面板、供应链管理、工艺优化分析、业务闭环中心、历史记录和个人中心。

项目还配置了测试和 Hook 校验。当前后端自动化测试覆盖 5 个智能体、编排器、RAG、业务闭环和接口集成。

## 3. 系统整体架构

现在我先讲整体架构。

用户在 Web 前端输入一个业务问题，比如“质量缺陷率上升，设备振动异常，需要优化工艺参数”。前端会把请求发送到后端统一接口。

后端首先进入 Agent Orchestrator，也就是智能体编排器。编排器会根据请求内容识别业务场景。如果只命中一个场景，就调用单个智能体；如果同时命中多个场景，就进入多智能体协同模式。

在执行过程中，编排器会调用 KnowledgeService 做知识检索，把命中的知识库证据补充到 evidence 字段中。每个智能体内部按自己的节点流程执行，比如输入校验、规则分析、风险判断、建议生成等。

智能体执行完成后，系统会生成统一返回结构，包括 `summary`、`decision`、`evidence`、`next_actions` 和 `node_feedback`。

其中顶层 `summary` 不是简单拼接流程日志。系统会通过业务答案质量门检查它是否过短、是否只有“已接收请求”或“请提供完整数据”等空洞内容。如果不合格，编排器会调用 BusinessAnswerService 生成专业业务答案，并在节点反馈里增加“答案综合”节点。这样无论是自动路由、指定智能体调用，还是多智能体协同，前端展示的主答案都更适合用户阅读。

最后，BusinessClosureService 会根据智能体结果自动聚合业务闭环对象，包括预警、工单、报告和行动项。这样前端不仅能看到一次分析结果，还能看到后续要追踪的业务对象。

## 4. 五个智能体总览

本项目一共实现了 5 个独立智能体。

第一个是生产调度优化智能体，负责生产计划、工单排程、产能评估和瓶颈分析。

第二个是质量检测与缺陷分析智能体，负责分析质检数据、识别缺陷模式，并按人机料法环追溯根因。

第三个是设备预测性维护智能体，负责分析振动、温度、电流等传感器数据，预测故障风险并生成维护工单。

第四个是供应链协同管理智能体，负责根据生产计划和库存数据生成采购建议、缺料预警、供应商评估和库存周转分析。

第五个是工艺参数优化智能体，负责分析温度、压力、速度、时间等工艺参数与产品质量之间的关系，推荐更优参数组合。

每个智能体都是独立模块，互不干扰，但可以通过编排器协同工作。

## 5. 生产调度优化智能体流程讲解

下面讲第一个智能体，生产调度优化智能体。

它的业务目标是：根据订单、工单、设备、工艺路线和交期要求，生成合理的排产方案，并提前发现产能瓶颈。

它的流程分为 6 个主要节点。

第一个节点是输入校验。系统会检查订单、工单、设备和工艺路线是否完整。如果缺少关键数据，流程会中止，并提示用户补充数据。

第二个节点是订单优先级排序。系统会根据订单交期、订单优先级和优化目标，对订单进行排序。比如高优先级订单和临近交期订单会优先排产。

第三个节点是工艺路线匹配。系统会为每个产品匹配加工路线。如果某个产品没有工艺路线，就会在证据中提示路线缺失。

第四个节点是产能评估与排程。系统会结合设备状态、工序工时和生产资源，生成初步排程结果。

第五个节点是瓶颈分析。系统会判断哪些设备利用率过高，哪些工序可能导致延期。

第六个节点是调度建议生成。系统会汇总排程结果，输出决策、证据和后续行动建议，比如调整交期、增加设备工时或启用替代设备。

这个智能体适合生产主管和计划员使用。

## 6. 质量检测与缺陷分析智能体流程讲解

第二个智能体是质量检测与缺陷分析智能体。

它的目标是自动分析质检批次、质量指标、缺陷记录和历史缺陷库，识别质量风险和缺陷根因。

它的流程包括 7 个节点。

第一个节点是输入校验。系统检查质检批次、检验指标和缺陷记录是否满足分析条件。

第二个节点是标准匹配。系统会把检测指标和质量标准进行匹配。如果标准缺失，会在节点反馈中提示。

第三个节点是指标评价。系统判断每个检测指标是否合格，比如尺寸、硬度、表面粗糙度是否超出标准范围。

第四个节点是缺陷分类。系统会按缺陷类型和严重度统计缺陷，比如划伤、裂纹、尺寸偏差等。

第五个节点是模式识别。系统会判断是否出现高频缺陷或者批量性缺陷模式。

第六个节点是根因追溯。系统会结合人、机、料、法、环，以及历史缺陷库，判断缺陷更可能来自操作、设备、材料、工艺还是环境。

第七个节点是改进建议生成。系统会根据分析结果给出隔离批次、复检、设备校准、材料追溯或工艺调整等建议。

这个智能体适合质量工程师和质量主管使用。

## 7. 设备预测性维护智能体流程讲解

第三个智能体是设备预测性维护智能体。

它的目标是通过传感器数据预测设备故障，提前生成维护建议，避免意外停机。

它的流程包括 9 个节点。

第一个节点是输入校验。系统会检查设备信息和传感器读数是否存在。

第二个节点是传感器状态评估。系统会分析振动、温度、电流等指标，判断是否正常、预警或严重异常。

第三个节点是趋势分析。系统不仅看单点数据，还会看一段时间内的变化趋势，比如振动值是否持续升高。

第四个节点是风险评估。系统会综合传感器异常、设备状态和历史信息，形成风险等级。

第五个节点是故障预测。系统会预测可能发生的故障类型，比如轴承磨损、电机过热或电流异常。

第六个节点是历史故障匹配。系统会从历史故障库中查找相似案例，提高预测可信度。

第七个节点是维护规程匹配。系统会匹配维护规则，确定工单类型、预计工时和维护动作。

第八个节点是故障时间窗口估计。系统会估算设备还可以运行多久，以及建议在哪个窗口维护。

第九个节点是维护工单生成。如果风险等级达到条件，系统会自动生成维护工单；如果风险较低，则建议继续监控。

这个智能体适合设备工程师和维修主管使用。

## 8. 供应链协同管理智能体流程讲解

第四个智能体是供应链协同管理智能体。

它的目标是根据生产计划、BOM、库存、采购单和供应商数据，降低缺料和库存积压风险。

它的流程包括 10 个节点。

第一个节点是输入校验。系统检查生产计划和库存数据是否存在。

第二个节点是物料需求分析。系统根据生产计划和 BOM 展开物料需求，计算每种物料需要多少。

第三个节点是库存风险评估。系统结合当前库存、可用库存和在途采购单，判断物料是否够用。

第四个节点是缺料风险分析。系统识别普通缺料和严重缺料。

第五个节点是积压分析。系统识别库存过高、周转偏低或超过安全上限的物料。

第六个节点是供应商评估。系统根据交期、质量和成本评价供应商风险。

第七个节点是采购建议生成。系统会推荐采购数量、推荐供应商和预计成本。

第八个节点是 PO 交付分析。系统检查在途采购单是否延期，以及延期对生产计划的影响。

第九个节点是协同行动生成。系统会生成采购、催交、替代供应商或生产计划调整建议。

第十个节点是库存周转分析。系统评估物料周转效率，帮助降低库存积压。

这个智能体适合采购、计划和供应链管理团队使用。

## 9. 工艺参数优化智能体流程讲解

第五个智能体是工艺参数优化智能体。

它的目标是分析历史生产数据和质量反馈，推荐更优的工艺参数组合，提高良品率和生产效率。

它的流程包括 8 个节点。

第一个节点是输入校验。系统会匹配工艺信息、产品信息、历史批次和参数约束。

第二个节点是历史数据分析。系统统计历史批次的良品率、缺陷率和生产节拍。

第三个节点是参数约束检查。系统检查当前温度、压力、速度、时间等参数是否超出约束范围，尤其关注安全关键参数。

第四个节点是质量相关性分析。系统分析参数变化和质量结果之间的关系，比如温度升高是否导致缺陷率上升。

第五个节点是最佳批次选择。系统根据优化目标选择表现最好的历史批次作为参考。

第六个节点是推荐参数生成。系统参考最佳批次和参数约束，推荐新的参数组合。

第七个节点是风险评估。系统评估参数越界、安全风险、目标冲突和历史数据不足等问题。

第八个节点是优化摘要生成。系统输出预期改善、推荐理由和后续验证建议。

这个智能体适合工艺工程师和持续改进团队使用。

## 10. 前端登录与注册演示

现在进入系统演示。

首先打开前端页面。可以看到登录页采用企业级安全登录界面，左侧是智能制造平台介绍，右侧是账号登录面板。

这里不在页面上展示默认账号密码，避免演示系统看起来不安全。实际演示时可以使用管理员账号登录，也可以点击注册创建一个新账号。

我先输入账号和密码，然后点击登录。登录成功后，系统进入运营总览页面。

如果点击注册，系统会进入注册页。输入用户名、邮箱和密码后，账号会保存在本地演示环境中。注册完成后可以直接进入系统，也可以返回登录页用新账号登录。

这个登录注册功能主要用于演示账号验证和会话保持。

登录后首先看到的是运营总览页面。

这个页面是管理层视角的统一入口，展示总调用次数、协同调用次数、预警总数和待处理事项。

页面中间有风险概览，可以看到最近的预警等级和来源智能体。

右侧或者下方可以看到智能体运行分布，用来说明哪个业务域近期调用较多。

下面还有最近工单和行动项。它们不是手工填写的，而是由智能体结果自动汇总形成的业务闭环对象。

## 12. 智能体工作台演示

接下来打开智能体工作台。

这里是智能体调用的核心入口。

上方可以选择智能体模式。如果不选择智能体，就是自动路由模式。系统会根据用户输入自动判断该调用哪个智能体。

如果选择某个具体智能体，就是指定调用模式。比如选择质量检测智能体，就会直接调用质量检测与缺陷分析流程。

我这里输入一个跨域问题：“质量缺陷率上升，设备振动传感器报警，需要优化工艺参数改善良品率”。

然后点击执行分析。

系统会识别到这是一个多智能体协同问题，因为里面同时包含质量、设备和工艺三个业务域。

执行完成后，可以看到返回结果中 `execution_mode` 是 collaborative，表示多智能体协同。

结果区域会展示智能分析结论、风险判断、建议处理步骤、分析依据和执行详情。对于知识图谱、预测性维护、机器人调度等方法型问题，前端会把答案整理成段落、步骤、公式块和分析路径图，避免出现一大段难读文本。

再往下看执行链路，可以看到质量智能体、设备智能体和工艺智能体是按顺序执行的。

节点反馈里可以看到每一步的状态和说明，这正是我们项目要求里“接口层可以看到节点式反馈”的体现。

业务闭环页签中可以看到系统自动生成的预警、工单、报告和行动项。

## 13. 单智能体业务页面演示

接下来演示单智能体业务页面。

先打开生产调度看板。这个页面默认绑定生产调度优化智能体。点击填入示例，再点击调用，就会执行生产调度分析。结果会展示排程风险、瓶颈设备和下一步建议。

这里需要说明的是：这些单智能体业务页面调用的是 `POST /api/v1/agents/{agent_name}/execute` 指定智能体接口。即使直接指定智能体，系统也会经过 Orchestrator 的质量门。如果原始摘要过短，系统会自动生成更完整的业务分析答案，并保留原始执行链路用于追溯。

然后打开质量检测中心。这个页面默认绑定质量检测与缺陷分析智能体。它适合演示缺陷分类、质量风险和根因定位。

再打开设备监控面板。这个页面默认绑定设备预测性维护智能体，可以演示传感器异常、故障预测和维护工单生成。

接着打开供应链管理页面。这个页面默认绑定供应链协同管理智能体，可以演示库存预警、采购建议和供应商风险。

最后打开工艺优化分析页面。这个页面默认绑定工艺参数优化智能体，可以演示参数越界、质量相关性和参数推荐。

这几个业务页面的结构一致，方便不同部门用户直接进入自己关心的业务模块。

## 14. 业务闭环中心演示

现在打开业务闭环中心。

这个页面汇总所有智能体产生的闭环对象。

第一个页签是预警，展示系统识别出的质量风险、设备风险、供应链风险和工艺风险。

第二个页签是工单，展示由智能体生成的维护工单、采购工单、质量处理工单或工艺优化工单。

第三个页签是报告，展示每次分析形成的业务报告，包括发现项和建议项。

第四个页签是行动项，展示后续需要人工执行或跟踪的事项。

业务闭环中心的价值是把智能体结果从一次性分析变成可追踪的管理对象。

## 15. 历史记录演示

接下来打开历史记录页面。

这里保存了每次智能体执行的请求、决策、摘要和完整结果。

点击任意一条历史记录，可以在右侧查看详情，包括原始请求、分析摘要、决策、执行模式、证据、节点反馈和业务闭环。

这说明系统具备可追溯性。录屏讲解时可以强调：每一次智能体调用都不是黑盒输出，而是可以回看流程和证据。

## 16. 后端接口演示

除了前端界面，本项目也支持后端接口调用。

统一执行入口是：

```text
POST /api/v1/agents/execute
```

这个接口支持自动路由。如果用户输入的问题只涉及一个业务域，系统会执行单智能体；如果涉及多个业务域，系统会执行多智能体协同。

指定智能体调用入口是：

```text
POST /api/v1/agents/{agent_name}/execute
```

比如生产调度智能体的名称是 `production_scheduling`，质量检测智能体的名称是 `quality_inspection`，设备维护智能体是 `predictive_maintenance`，供应链智能体是 `supply_chain_management`，工艺优化智能体是 `process_parameter_optimization`。

这两个接口最终都会进入 AgentOrchestrator。区别是统一入口会自动识别业务场景，指定入口会直接使用路径里的智能体名称。两种方式都会返回统一结构，并且都会经过业务答案质量门。

接口返回中最重要的字段是：

`summary` 表示分析摘要。

`decision` 表示系统决策。

`evidence` 表示证据来源。

`next_actions` 表示建议后续行动。

`node_feedback` 表示节点式执行反馈。

如果本次触发了业务答案增强，`node_feedback` 中会出现一个名称为“答案综合”的节点，表示系统已经将流程摘要提升为可直接阅读的技术方案或业务分析结论。

`closure` 表示业务闭环对象，包括预警、工单、报告和行动项。

## 17. 项目工程化说明

从工程结构上看，后端代码分层比较清晰。

`app/api` 是接口层，只负责入参校验和编排调用。

`app/agents` 是智能体层，包含 5 个业务智能体。

`app/services` 是服务层，包含业务计算、编排器、知识检索、业务答案合成和业务闭环服务。

`app/schemas` 是数据模型层，定义每个智能体的输入输出结构。

`app/db` 是数据库连接层，通过配置构造 MySQL 连接。

`frontend` 是 Vue3 前端工程。

`tests` 是自动化测试。

`docs` 是项目文档，包括流程图和讲解材料。

项目遵守一个统一原则：接口层不堆业务逻辑，复杂逻辑放到 services 或 agents，方便维护和扩展。

## 18. 测试与验收说明

项目可以通过以下命令进行验收。

首先运行 Hook 校验：

```powershell
python hooks\check_claude.py
```

这个脚本会扫描敏感模块和禁止模式，避免把密钥、危险代码或不合规内容提交进去。

然后运行后端测试：

```powershell
python -m pytest
```

当前测试覆盖业务闭环、护栏、集成测试、知识检索、编排器和 5 个智能体。

本项目还专门补充了业务答案质量测试，验证五个独立智能体指定调用不会再返回空洞摘要，并且会包含“答案综合”节点。最近一次拆分验收中，`test_orchestrator.py` 285 个用例通过，五个智能体模块 269 个用例通过，业务答案服务 19 个用例通过，护栏、闭环、RAG 和集成测试合计 109 个用例通过。

最后运行前端构建：

```powershell
cd frontend
npm run build
```

如果这些命令都通过，就说明项目具备基本交付条件。

前端构建时如果看到主包体积超过 500KB 的提示，这是当前版本的性能优化项，不影响功能演示。后续可以通过路由懒加载和 manualChunks 做拆包优化。

## 19. 项目总结

最后总结一下。

本项目实现了一个面向智能制造场景的多智能体协作平台。

业务上，它覆盖生产调度、质量检测、设备维护、供应链管理和工艺优化 5 个关键工厂管理场景。

技术上，它使用 FastAPI、MySQL、LangChain、Chroma、Vue3 和 Element Plus，具备后端接口、前端界面、RAG 检索、多智能体编排、业务闭环和自动化测试。

流程上，每个智能体都有清晰的节点反馈，可以在接口层和前端页面看到每一步执行结果。

展示上，前端不仅能完成调用，还能展示运营总览、智能体工作台、业务闭环中心和历史记录，适合录屏演示和项目答辩。
